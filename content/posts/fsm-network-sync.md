# 状态机与网络相关问题 
## 架构背景

在我询问状态机是否过时的时候，老程序给出的回答：


如果你口中的“状态机”是指在一个 `Tick` 函数里塞满几千行的 `switch-case`，或者满屏的 `if-else` 来切换 `enum`，那不叫状态机，那叫**屎山代码**，这种写法在十年前就已经被淘汰了。但如果你指的是有限状态机（FSM）、分层状态机（HSM）或是下推自动机（PDA）的数学模型与工程抽象，我明确告诉你：**它们是现代复杂游戏架构的绝对基石。**

## **极端场景问题：**

假设你用 C++ 写了一个严格的强联网角色状态机。由于网络环境恶劣（UDP丢包严重），服务器下发的“状态A切入确认包”比下一个“基于状态A的攻击动作 RPC”**晚到了 200 毫秒**。此时你的客户端正在另一个状态里，强行执行 RPC 会导致崩溃或表现撕裂。

## 💬回答

对RPC添加状态依赖标记，以及在客户端实现消息队列，在发现消息队列依赖某些状态的时候但是此时没有此状态的时候就添加到队列中，等待网络同步状态包。

## **😡追问**

仅仅入门摸到了消息队列的皮毛，在**工业级强联网的底层架构**中活不过一次**弱网测试**，因为会出现以下问题：

### 1. 内存以及相关参数生命周期处理

1.   只考虑了包晚到问题，如果包没到呢？
2.  是不是就一直在队列中缓存消息？
3.  消息队列的大小你设置的多大呢？
4.  什么情况下清空消息队列呢？
5.  你的**队列生命周期**（TTL）该怎么设计？
6.  当确认包确认丢失的时候 （比如收到了更高等级序列号的包）时，队列中失去前置条件的孤儿RPC怎么处理呢？怎么安全地丢弃和进行内存回收，而不引发野指针异常？

### 2. 还有表现层的撕裂

1. 假如200ms之后你执行了攻击动作，但是此时别人已经跑远了
2. 此时还需要执行攻击动作吗？

### 3. 被动等待的架构劣势

1. 如果客户端真的要做到客户端预测和回滚，客户端是不是在本地立刻预测切入状态A，并直接生成攻击的RPC，同时把带有序列号的操作发给服务器呢？
2. 是不是本地内存需要维护一个环形缓冲区呢用于保存过去200ms的实体历史快照呢？
3. 如果服务端回包判定预测失效了，如何利用缓冲区在一帧内强行回滚Rollback并重新模拟呢

### 🤔疑问
虚幻引擎自带的RPC结构对弱网环境有处理吗？ 
**答**：没有，虚幻引擎原生RPC仅提供基础的可靠/不可靠传输能力，完全不处理丢包、RPC过期、被动延迟等问题

### 问题1🚗

问题本质：状态确认包彻底丢失 -> 客户端队列无限缓存RPC->内存思索 ； 裸指针会引发野指针崩溃

之前考虑RPC队列只考虑了非环形队列，我们可以采用**环形队列**进行处理。

1. **队列TTL生命周期：动态RTT（往返时间）自适应，拒绝固定值**
	不能够写死200ms,用UE原生提供的网络RTT计算 ：TTL =3* 玩家当前的RTT （覆盖三次超时重传） ，同时增加上下限，避免网络异常时TTL失效。
2. 孤儿RPC判定：序列号单调递增 = 丢包证据  【前提得是TCP,因为UDP天然乱序的】
	服务器分配的全局单调递增序列号 (S1 , S2 , S3) 客户端一但收到**S'>已确认的最大序列号**，直接判定 `S -S'`的确认包彻底丢失，队列中所有依赖这些序列号的RPC都是孤儿RPC，直接回收

	可以修改为如下操作：
	核心设计：抗抖动缓冲区 (Jitter Buffer) + 滑动窗口 (Sliding Window)
	1. **不看最大序列号就删消息**！所有包先放进一个**固定时间 / 帧数的缓冲区**；
	2. 设置**容忍窗口**（动作游戏标准：3 帧 / 20~30ms）；
	3. 窗口时间内，等待乱序的包自动归位；
	4. **超过窗口时间还没到**，才判定为真・丢包，再清理孤儿 RPC；
	5. 窗口内的乱序包，自动按序列号排序后再处理。
	
	就像**排队检票**：
	- 本来应该 1、2、3 号排队进场；
	- 结果 3 号先跑过来了，不能直接把 2 号拉黑；
	- 等**30 毫秒**，2 号大概率就赶来了；
	- 等了半天 2 号还没来，才判定 2 号丢了，再清理。



3. ~~安全内存回收：UE智能指针 + 固定环形队列~~
	~~杜绝使用裸指针，使用TSharedPtr 管理RPC数据，出队自动释放内存；队列使用固定大小的环形缓冲区，从根源杜绝无限膨胀。~~  

	3的这段纯错了，这对内存简直是巨大的打击！！ 性能杀手！ 高频的RPC用智能指针会触发堆内存分配，原子引用计数，内存碎片，CPU缓存未命中等，低端机直接帧率爆炸。
	
	**⚠️这个涉设计完全违背了工业级设计原则**
	**✅ 正确方案（UE / 游戏行业唯一标准）
	核心设计：预分配连续内存 + 无指针 + 游标环形队列
		1. **提前划一块固定大小的连续内存**（数组 / 静态缓冲区），比如直接预分配 64/128 格空间；
		2. **里面只存 轻量级 UStruct / 原生结构体**（无继承、无指针、纯数据）；
		3. 用**读写游标（Head/Tail 数字索引）** 控制位置，满了直接**覆写旧数据**；
		4. **全程无 new、无 delete、无智能指针、无内存分配**，CPU 缓存命中拉满，零开销。
	 
4. **清空消息队列的场景**
	1. 网络断开、重连失败 ：客户端掉线的时候，网络卡死的时候都要清空，因为断线之后的队列中的消息永远发不出去，留着占内存
	2. 客户端预测错误，服务器强制回滚的时候 ->全量清空，以为内服务器说客户端的状态错误，那就需回滚一瞬间清空客户端的队列
	- **掉线重连失败 → 全清**
	- **切服切场景 → 全清**
	- **预测错了回滚 → 全清**
	- **消息超时 → 清过期**
	- **服务器新确认 → 清孤儿**
	- **被控 / 死亡 → 清操作**
