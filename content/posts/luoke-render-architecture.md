要理解这些优化，我们首先需要补充一个移动端 GPU 硬件架构的底层基础知识：**TBDR（基于图块的延迟渲染）**。理解了它，你就能彻底明白为什么要“合并 Pass”以及“减少显存写回”。

---

### 一、 核心概念：什么是“减少显存写回”？（对应图 1）

在移动端（手机）上，GPU 的架构和 PC 有很大不同。为了省电和控制发热，移动端 GPU 使用了 **TBDR (Tile-Based Deferred Rendering)** 架构。

- **Tile Memory（图块内存 / 片上内存）：** GPU 内部的一块极小、但速度极快的缓存。你可以把它理解为 CPU 的 L1 缓存。在这里读写数据几乎不消耗带宽和电量。
    
- **System Memory（系统内存 / 显存）：** 手机的物理内存。容量大，但读写速度慢，且**极度消耗带宽和电量**（发热的元凶）。
    

**传统的渲染流程（未合并 Pass）：**

1. GPU 把屏幕切成很多个小方块（Tile）。
    
2. 在 Tile Memory 中画完第一个 Pass（比如算完半透明）。
    
3. **写回（Store/Resolve）：** 把 Tile Memory 里的结果写到 System Memory 中保存起来。
    
4. 进行下一个 Pass 时，再从 System Memory 把数据**读取（Load）**回 Tile Memory 接着画。
    

**合并 Pass 的意义（SubPass 技术）：** 图 1 中展示的优化，核心思想就是：**“既然数据都在 Tile Memory 里，那我们就一口气把这几个操作（颜色分级、半透明、水面扭曲等）全算完，最后再统一写回 System Memory。”**

- **结果：** 将 5 个 Pass 合并成 2 个 Pass，省去了中间 3 次往返 System Memory 的“写回”和“读取”操作。这就是 PPT 中提到的“减少了三次 GPU 显存写回”，直接大幅降低了带宽占用（图 2 中带宽下降 30%）。

### 二、 标准渲染管线：从 PrePass 到 Tonemapping 都在干嘛？（对应图 4）

在了解极致优化前，我们先看看图 4 中标准的渲染管线（按顺序）：

1. **Prepass & Basepass（预渲染与基础渲染）：**
    
    - **干什么：** 这是最基础的绘制。算出屏幕上每个像素对应哪个模型、什么材质（颜色、粗糙度、法线等），以及它们的深度（离相机的距离）。
        
2. **SSAO Compute（屏幕空间环境光遮蔽）：**
    
    - **干什么：** 通过分析深度图，找出场景中的“死角”（比如墙角、草丛根部），让这些地方变暗。这能极大增强画面的立体感。
        
3. **SSR Pass（屏幕空间反射）：**
    
    - **干什么：** 计算光滑表面（如水面、湿润的地面、金属）对屏幕内其他物体的实时反射。
        
4. **SubPass Chain（子通道链）：**
    
    - **干什么：** 处理那些不需要打断渲染流程的附加效果，比如半透明物体（玻璃、普通的特效）、贴花等。
        
5. **Bloom（泛光/辉光）：**
    
    - **干什么：** 提取画面中特别亮的部分（比如太阳、发光的武器特效），进行模糊处理，然后再叠加回画面上，产生一种“光芒四射”的视觉效果。
        
6. **Tonemapping（色调映射）：**
    
    - **干什么：** 游戏引擎内部是用 HDR（高动态范围，亮度可以远远超过 1）计算的，但手机屏幕只能显示 LDR（0 到 1 的亮度）。Tonemapping 就像照相机的曝光调整，把 HDR 的超广颜色和亮度范围，压缩/映射到屏幕能显示的色彩空间中，同时进行最终的颜色校正（Color Grading）。

### 三、 How to One Pass：低配 HDR 管线去掉了什么？为什么？（对应图 5 和 图 6）

图 5 提出终极目标：**One Pass（一镜到底）**。也就是从画基础模型开始，直到最后输出到屏幕，中间绝不把数据写回 System Memory。

为了实现这个近乎疯狂的性能压榨（专门针对低配手机），**必须砍掉 SSAO、SSR 和 Bloom**。

**为什么必须砍掉它们？（这是理解渲染管线的关键）**

在 TBDR 架构的 Tile Memory 中，GPU 是一个一个小方块（比如 32x32 像素）独立处理的。当前正在处理的方块，**无法知道旁边方块的数据**。

- **SSAO 和 SSR 的致命伤：** 这两个效果叫“屏幕空间（Screen Space）”效果。比如计算水面反射（SSR），水面的像素需要知道屏幕边缘天空的像素是什么颜色，才能反射出来。这就要求 GPU 必须统揽全局（拿到完整的深度图和颜色图）。所以，**必须把之前的计算结果写回 System Memory**，然后再把整张图拿来做跨像素的采样。这就打断了 One Pass。
    
- **Bloom 的致命伤：** 泛光需要把高亮区域提取出来，然后进行“降采样”和“高斯模糊”（让光晕晕染开来）。模糊操作也是一种跨像素的运算（当前像素会受到周围几十个像素的影响），在极小的 Tile Memory 里根本做不了，必须写回 System Memory。
    

**低配 HDR 管线的最终实现：** 如图 5 所示，通过忍痛割爱去掉了上述三个依赖“全局像素信息”的特效，剩下的 `Prepass Basepass` -> `SubPass Chain` (半透明等) -> `Tonemapping` (色调映射、雾效 Fog) 这些全部都是**“点对点”（单像素自己就能算完）**的操作。

所以，它们可以完美地打包在一起，全程在高速的 Tile Memory 中闭环计算。这不仅保住了基本的 HDR 渲染流程和色调质感（图 6 对比画面仍然不错），还在低端机（如骁龙 730G）上砍掉了高达 30.9% 的显存写入带宽，彻底释放了性能。
